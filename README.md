# Batmanuel

**Batmanuel** is a code quality platform built with NestJS. It analyzes repositories to measure code quality, detect security vulnerabilities, flag outdated or vulnerable dependencies, and identify duplicated code. It exposes an API consumed by a CLI and integrates natively with CI/CD pipelines (GitHub Actions initially).

This repository currently ships with **mocked routes**, so you can validate the full integration flow (token generation, authenticated calls, pipeline decisions) before the real analysis engines are implemented.

## Architecture Overview

Batmanuel is designed as an orchestrator rather than a single monolithic analyzer. Instead of reimplementing static analysis from scratch, it is meant to:

1. Receive analysis reports generated by external tools (linters, SAST scanners, dependency scanners).
2. Normalize those reports into a common `AnalysisReport` model.
3. Calculate a consolidated quality/security score per project, branch, and commit.
4. Expose that data through a REST API, consumed by a CLI, CI/CD pipelines, and (later) an Angular dashboard.

```
CLI / GitHub Actions  --->  Batmanuel API (NestJS)  --->  Normalized report + score
                                   |
                                   v
                          (future) PostgreSQL storage
                                   |
                                   v
                          (future) Angular dashboard
```

## Getting Started

### Prerequisites

- Node.js 18+
- npm

### Installation

```bash
npm install
```

### Running locally

```bash
npm run start:dev
```

The API will be available at `http://localhost:3000`.

### Testing a request

```bash
curl -X POST http://localhost:3000/analyze \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer MOCK-TOKEN-1234567890" \
  -d '{"projectId":"sindigo-backend","branch":"main","commit":"abc123"}'
```

## API Routes

### POST /auth/token

Generates an access token for a given project. Currently returns a **fixed mock token**, used only to validate the authentication flow end-to-end.

**Request body:**

```json
{
  "projectId": "sindigo-backend"
}
```

**Response:**

```json
{
  "projectId": "sindigo-backend",
  "token": "MOCK-TOKEN-1234567890",
  "expiresIn": "30d"
}
```

> In the real implementation, this endpoint will issue a signed JWT tied to a `projectId`/`clientId`, with support for revocation and expiration.

---

### POST /analyze

Receives project/branch/commit information and returns an analysis report. Requires authentication (see [Authentication](#authentication)).

**Request body:**

```json
{
  "projectId": "sindigo-backend",
  "branch": "main",
  "commit": "abc123"
}
```

**Response:**

```json
{
  "projectId": "sindigo-backend",
  "branch": "main",
  "commit": "abc123",
  "timestamp": "2026-07-15T15:00:00.000Z",
  "score": 82,
  "metrics": {
    "securityCritical": 1,
    "securityHigh": 2,
    "securityMedium": 5,
    "qualitySmells": 12,
    "duplications": 3,
    "outdatedDeps": 2
  },
  "issues": [
    {
      "engine": "mock-engine",
      "type": "security",
      "severity": "critical",
      "ruleId": "sql-injection",
      "message": "Possible SQL injection (mock example)",
      "file": "src/users/users.service.ts",
      "line": 42
    }
  ]
}
```

| Field | Type | Description |
|---|---|---|
| score | number | Overall quality/security score (0-100) |
| metrics.securityCritical/High/Medium | number | Count of security issues by severity |
| metrics.qualitySmells | number | Count of code smells detected |
| metrics.duplications | number | Count of duplicated code blocks |
| metrics.outdatedDeps | number | Count of outdated/vulnerable dependencies |
| issues[].type | string | `security`, `quality`, or `dependency` |
| issues[].severity | string | `critical`, `high`, `medium`, `low`, or `info` |

---

### GET /projects/:id/summary

Returns a historical summary for a given project, including score trend across recent analyses. Requires authentication.

**Response:**

```json
{
  "projectId": "sindigo-backend",
  "lastScore": 82,
  "trend": [76, 78, 80, 82],
  "lastAnalysisAt": "2026-07-15T15:00:00.000Z"
}
```

## Authentication

All routes except `POST /auth/token` require a Bearer token in the `Authorization` header:

```
Authorization: Bearer MOCK-TOKEN-1234567890
```

The `TokenGuard` (in `src/auth/token.guard.ts`) validates this header. In the current mock phase, only the fixed token `MOCK-TOKEN-1234567890` is accepted.

**Planned evolution:**

- Replace the fixed mock token with signed JWTs (`@nestjs/jwt` + `passport-jwt`), following the same pattern already used in other NestJS projects (custom `AuthGuard('jwt')`).
- Each token will carry a `projectId`/`clientId` claim to scope requests to a specific project.
- Tokens will be generated and revocable via the future Angular admin interface.

## GitHub Actions Integration

A sample workflow is provided at `.github/workflows/code-quality.yml`. It:

1. Checks out the repository.
2. Builds a report payload with project, branch, and commit information.
3. Sends the payload to `POST /analyze`, authenticating with a Bearer token.
4. Reads the returned `score` and fails the pipeline if it falls below a defined threshold.

**Required repository secrets:**

| Secret | Description |
|---|---|
| `QUALITY_API_URL` | Public URL of the Batmanuel API (e.g., via a deployed instance or a tunnel like ngrok during testing) |
| `QUALITY_API_TOKEN` | Bearer token used to authenticate requests (use `MOCK-TOKEN-1234567890` during the mock phase) |

**Example step:**

```yaml
- name: Send report to Batmanuel
  run: |
    curl -s -X POST "$QUALITY_API_URL/analyze" \
      -H "Content-Type: application/json" \
      -H "Authorization: Bearer $QUALITY_API_TOKEN" \
      -d @report.json \
      -o response.json
  env:
    QUALITY_API_URL: ${{ secrets.QUALITY_API_URL }}
    QUALITY_API_TOKEN: ${{ secrets.QUALITY_API_TOKEN }}
```

## Roadmap: From Mock to Real Engines

The current version is intentionally mock-only, to let you validate the integration flow (token → authenticated request → response → pipeline decision) without any real analysis logic. Planned next steps:

1. **Authentication**: replace the fixed mock token with real JWT issuance and validation.
2. **Analysis engines**: introduce an `EnginesModule` with adapters for tools such as:
   - ESLint (code smells, complexity, basic duplication)
   - Semgrep (security-focused static analysis)
   - Dependency/CVE scanners (outdated or vulnerable libraries)
3. **Normalization**: add a `NormalizationModule` to convert each engine's output into the shared `Issue`/`AnalysisReport` model.
4. **Rules and scoring**: add a `RulesModule` to compute the final score and apply pass/fail thresholds.
5. **Persistence**: store historical reports in PostgreSQL for trend analysis.
6. **Dashboard**: build the Angular frontend consuming `GET /projects/:id/summary` and related endpoints.
7. **CLI**: ship a standalone CLI (`npx batmanuel analyze`) that collects local reports and pushes them to the API, simplifying pipeline usage.
